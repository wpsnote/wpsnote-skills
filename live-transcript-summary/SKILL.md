# 边听边总结 Skill

实时监听当前 WPS 笔记中的音频录音转写，每 60 秒轮询一次，根据场景自动选择模板整理内容并写回笔记。

## 核心工作流

```
启动时（仅一次）:
  0. 学习用户总结风格（search_notes 找历史总结笔记）
  0. 向用户发送启动告知

loop:
  1. get_current_note()                    → 获取当前笔记 note_id
  2. get_note_outline(note_id)             → 扫描 NoteAudioCard blocks
  3. get_audio_transcript(shorthand_id)   → 获取转写内容
  3.5 推断发言人真实身份（结合笔记内容 + 转写线索）
  4. 场景识别 → 选择模板 → 生成摘要（参考用户风格）
  5. 检测笔记中的 *** → 结合文风补全内容
  6. 写回笔记（insert 或 replace）
  7. sleep 60 秒 → 回到步骤 1
```

## 启动流程

### 阶段 0：学习用户风格（仅在启动时执行一次）

在第一轮正式轮询前，先搜索用户历史总结笔记，学习其写作风格：

```
search_notes({ keyword: "总结 纪要 摘要", limit: 5, sort: "update_time", direction: "desc" })
→ 找到近期有总结性内容的笔记
→ read_note(note_id) 读取 1-2 篇
→ 提取风格特征（见「风格学习」章节）
```

若未找到历史总结，使用默认模板风格，无需告知用户。

### 阶段 0.5：启动告知

学习完风格后，立即向用户输出启动告知，格式如下：

```
已开始边听边总结 ✓

当前笔记：[笔记标题]
识别场景：[场景类型]（如无法识别则显示「等待录音开始后自动识别」）
参考风格：[找到/未找到历史总结，找到则显示参考了哪篇笔记]
轮询间隔：60 秒

---
如需 AI 援助，可以在笔记中这样做：

  ① 补全内容：在任意位置输入 ***
     AI 会结合录音内容和上下文自动补全
     例：「今天讨论的核心问题是 ***」

  ② 随时停止：告诉我「停止总结」即可
---
第一轮将在 60 秒后开始，或录音有新内容时立即触发。
```

### 启动方式（定时循环）

```bash
# 用 shell sleep 实现定时轮询
while true; do
  # 执行一轮总结（通过 MCP 工具调用）
  # ...
  sleep 60
done
```

**实际执行时**：每轮通过 MCP 工具完成笔记读取 + 转写获取 + 内容生成 + 写回，然后 `sleep 60` 后继续下一轮。

## 步骤详解

### Step 1：获取当前笔记

```
get_current_note()
→ { note_id, title }
```

若返回 `NO_ACTIVE_EDITOR_WINDOW`，提示用户打开笔记后重试。

### Step 2：扫描音频卡片

```
get_note_outline(note_id)
→ 找出所有 type="note_audio_card" 的 block
→ 提取 attrs.shorthand_id、attrs.status、attrs.title
```

**状态判断**：
- `status="recording"` 或 `status="paused"` → 正在录制，继续获取转写
- `status="stopped"` → 录制已结束，仍可获取完整转写
- 无音频卡片 → 跳过本轮，输出提示「当前笔记未检测到录音」

### Step 3：获取转写内容

```
get_audio_transcript(shorthand_id)
→ { sentences: [{ text, speaker, start_time, end_time }] }
```

将 sentences 拼接为完整文本，保留时间戳信息供模板使用。

### Step 3.5：发言人身份推断

转写系统通常将说话人标记为「发言人 1」「发言人 2」等编号，需要在总结前尝试推断真实身份。

**推断来源（按优先级）**：

1. **笔记正文**：读取笔记中已有的文字内容，查找人名、职位、公司名等信息，结合出现位置判断对应关系（如笔记标题写了「与 XX 的访谈」，则可能对应某个发言人）
2. **转写内容本身**：
   - 自我介绍：「我是 XX」「我叫 XX」「My name is XX」
   - 称呼对方：「XX 你觉得」「请问 XX」→ 反推另一方身份
   - 头衔/角色：「作为产品经理」「我们公司」→ 结合上下文推断
   - 专有名词：某人反复提及特定项目/产品/团队 → 结合笔记内容匹配
3. **笔记标题和标签**：如标题含人名、活动名，可辅助推断

**推断结果处理**：

| 情况 | 处理方式 |
|------|---------|
| 确定是某人（转写中有明确自我介绍） | 直接使用真名 |
| 较有把握（80% 以上，有间接证据） | 使用真名，标注「❓待确认」 |
| 不确定 | 保留「发言人 1 / 2」编号，不猜测 |

**多轮推断与纠错**：每轮轮询时重新检查推断结果。若新的转写内容出现了更强的线索（如明确自我介绍），且与之前的推断不一致，则：

1. 确认新推断更可靠
2. 用 `search_note_content` 在笔记中搜索旧名称的所有出现位置
3. 用 `batch_edit` 批量替换为正确名称（包括带 `❓待确认` 标注的版本）
4. 在本轮输出中告知用户：「已将「[旧名]」更正为「[新名]」，共替换 N 处」

### Step 4：场景识别

根据笔记标题、已有内容关键词、转写内容特征，判断场景类型：

| 场景关键词 | 识别到的场景 | 使用模板 |
|-----------|------------|---------|
| 会议、讨论、周会、评审 | 会议记录 | 会议纪要模板 |
| 课堂、授课、教学、老师 | 课堂笔记 | 课堂笔记模板 |
| 培训、讲师、学员 | 培训课程 | 培训课程模板 |
| 知识分享、分享会、经验分享 | 知识分享 | 知识分享模板 |
| 直播、主播、观众 | 直播内容 | 直播记录模板 |
| 播客、嘉宾、节目 | 播客/视频 | 播客纪要模板 |
| 采访、记者、被采访 | 采访记录 | 采访记录模板 |
| 新闻、报道、媒体 | 新闻采访 | 新闻采访模板 |
| 谈判、甲方、乙方、合同 | 商务谈判 | 商务谈判模板 |
| 复盘、总结、里程碑 | 项目复盘 | 项目复盘模板 |
| 庭审、原告、被告、法庭 | 庭审记录 | 庭审纪要模板 |
| 患者、诊断、医嘱、症状 | 病例记录 | 病例记录模板 |
| 口述、文章、博客 | 口述转文章 | 文章模板 |
| 电话、通话、客服 | 电话录音 | 电话记录模板 |
| 无明显特征 | 通用场景 | 通用转写模板 |

### Step 5：检测 *** 补全请求

扫描笔记内容，识别用户在正文中输入的 `***`（补全占位符）：

**识别规则**：
- `***` 出现在普通段落、标题、列表项中 → 需要补全
- `***` 出现在代码块（`<codeblock>`）内 → **跳过，不处理**
- `***` 出现在数学公式、正则表达式等逻辑内容中 → **跳过，不处理**

**补全逻辑**：
1. 读取 `***` 所在 block 的上下文（前后各 5 个 block）
2. 结合当前音频转写内容
3. **分析上下文文风**（见「文风匹配」章节），用相同风格生成补全内容
4. 替换 `***`，保持句子自然流畅

**文风匹配**：补全时需分析 `***` 周围文本的写作风格，并严格模仿：

| 上下文特征 | 匹配策略 |
|-----------|---------|
| 口语化、第一人称（「我觉得」「感觉」「其实」） | 补全内容保持口语风格，不刻意书面化 |
| 书面化、客观陈述（「本次会议」「经讨论决定」） | 补全内容使用正式书面语 |
| 简短碎片化（每条 10 字以内） | 补全内容同样简短，不扩展 |
| 详细展开（每条 50 字以上） | 补全内容同样展开说明 |
| 带标点习惯（如惯用「、」「；」） | 补全内容沿用相同标点习惯 |
| 中英混写（如「Q2 的 growth」） | 补全内容保持相同混写风格 |

示例：
- 用户写（口语风格）：`今天聊的核心其实就是 ***，感觉挺有意思的`
- 转写内容提到：「我们今天主要讨论 Q2 的增长策略」
- 补全结果：`今天聊的核心其实就是 Q2 增长策略怎么做，感觉挺有意思的`

- 用户写（书面风格）：`本次会议的核心议题为 ***`
- 转写内容提到：「我们今天主要讨论 Q2 的增长策略」
- 补全结果：`本次会议的核心议题为 Q2 增长策略的制定与执行`

### Step 6：生成摘要内容

**重要约束**：
- 严格基于转写内容，不添加转写中不存在的信息
- 跳过空内容（无行动项就不写行动项章节）
- 避免发散，聚焦转写本身
- 模糊信息标注「待确认」
- **参考用户风格**（见「风格学习」章节），用用户习惯的方式组织内容
- **默认使用美观排版**（见「排版规范」章节），除非用户明确要求纯文本

**增量更新策略**：
- 首次运行：在笔记末尾插入完整摘要
- 后续轮次：检测已有摘要 block（通过特征标题识别），更新内容而非重复插入
- 新增内容用 `insert` 追加，已有内容用 `replace` 更新
- **待办事项（Todo list）必须始终放在摘要最末尾**：追加新章节时，用 batch_edit 先 delete 待办块，再在新内容之后重新 insert 待办块，保证顺序正确

### Step 7：写回笔记

```
get_note_outline(note_id)    → 刷新 block ID（写入前必须刷新）
batch_edit(note_id, [...])   → 原子写入摘要内容
```

写入完成后，输出本轮摘要的简短说明（字数、主要内容），然后等待 60 秒进入下一轮。

## 风格学习

启动时通过搜索历史笔记，提取用户的总结习惯，用于指导本轮摘要生成。

### 搜索策略

```
search_notes({ keyword: "总结 纪要 摘要 会议 复盘", limit: 5, sort: "update_time", direction: "desc" })
```

若无结果，尝试：
```
search_notes({ keyword: "要点 核心 结论", limit: 3 })
```

### 提取维度

读取 1-2 篇历史总结笔记后，分析以下维度：

| 维度 | 观察点 | 示例 |
|------|-------|------|
| **语言风格** | 口语 vs 书面，第一人称 vs 客观陈述 | 「我觉得」vs「经讨论」 |
| **结构偏好** | 扁平列表 vs 层级标题，是否用表格 | 全用 bullet vs 分章节 |
| **详略程度** | 每条多少字，是否展开说明 | 5 字 vs 50 字 |
| **标点习惯** | 常用标点，是否中英混写 | 惯用「；」分隔 |
| **信息取舍** | 倾向保留哪类信息（结论/过程/数据/人名） | 只记结论不记过程 |
| **特殊格式** | 是否用高亮块、引用块、特定 emoji | 惯用 `>` 引用重要句子 |

### 应用方式

提取风格特征后，在生成摘要时：
- 调整语言风格（口语/书面）
- 匹配结构偏好（扁平/层级）
- 控制详略程度（简短/展开）
- 沿用标点和混写习惯
- 优先保留用户偏好的信息类型

**注意**：风格学习是参考，不是复制。转写内容决定信息，用户风格决定表达方式。

## 排版规范

默认使用富文本排版，让摘要在 WPS 笔记中视觉清晰、层次分明。用户明确说「纯文本」「不要排版」时才退回到普通段落。

### 核心排版元素

**① 单栏 columns（带色块）**

用于承载「一个发言人的完整观点」或「一个需要视觉区分的主题块」。用单栏 `columns` 而非 `highlightBlock`——内部不受限制，可以放任意 block（标题、列表、引用、代码块等）。

颜色语义（使用 `columnBackgroundColor`）：
- 蓝色 `#EBF2FF`：中性信息、发言人观点
- 黄色 `#FFF5EB`：注意事项、待确认
- 绿色 `#E8FCEF`：结论、共识
- 红色 `#FFECEB`：风险、问题、争议点
- 紫色 `#FAF0FF`：补充信息、延伸内容

```xml
<columns>
  <column columnBackgroundColor="#EBF2FF">
    <h4>发言人 · 身份信息</h4>
    <p listType="bullet" listLevel="0">要点一</p>
    <p listType="bullet" listLevel="0">要点二</p>
    <blockquote>「原话片段」</blockquote>
  </column>
</columns>
```

**② blockquote（引用块）**

放在色块内部，紧跟相关要点之后，保留有价值的原话。选择能代表观点的 1-2 句，不要把转写全部引用。

**③ 颜色强调**

仅用于行内关键词、数据、专有名词，不要大段上色：

```xml
<span fontHighlightColor="#FBF5B3">关键数据</span>
<span fontColor="#C21C13">风险词</span>
```

**④ 双栏 columns（对比布局）**

仅用于内容天然对立的场景（如谈判双方、优缺点对比）：

```xml
<columns>
  <column columnBackgroundColor="#EBF2FF">
    <h4>甲方 / 正方</h4>
    <p listType="bullet" listLevel="0">内容</p>
  </column>
  <column columnBackgroundColor="#FFECEB">
    <h4>乙方 / 反方</h4>
    <p listType="bullet" listLevel="0">内容</p>
  </column>
</columns>
```

### 色块使用判断

**应该用色块的情况**：
- 有明确发言人，且该人有 2 条以上观点
- 一个完整主题/议题，内容超过 3 条要点
- 需要视觉区分的并列内容（多个人、多个方案）

**不应该用色块的情况**：
- 只有 1 条信息
- 基本信息、时间地点等元数据
- 行动项/Todo 列表（直接用 todo list）
- 内容少、结构简单时（普通段落 + bullet list 更清晰）

### 结构组织原则

内容先按**主题/议题**分章节（`h2`/`h3`），再在章节内用色块承载各发言人或子议题。不要跳过主题层级直接堆色块。

```
h2 主题一
  ├── columns(单栏蓝) 发言人 A（含要点 + 引用）
  ├── columns(单栏蓝) 发言人 B（含要点 + 引用）
  └── columns(单栏绿) 结论/共识

h2 主题二
  ├── 普通段落（内容少时不用色块）
  └── columns(单栏绿) 关键决策

h2 行动项
  └── todo list（不套色块）
```

### 排版原则

1. **色块是容器**：一个色块承载一个人或一个完整主题，内部可放任意内容
2. **先有主题，再有色块**：用标题组织章节，色块在章节内使用
3. **颜色有语义**：蓝=观点，绿=结论，黄=注意，红=风险，不随意换色
4. **引用要筛选**：只保留能代表观点的原话
5. **简单内容不装饰**：内容少时普通段落比堆色块更清晰

## 场景模板

各模板均遵循「严格基于转写内容、跳过空内容、默认美观排版」原则。

### 通用转写模板

```xml
<h2>转写摘要</h2>
<p><strong>时间</strong>：[日期时间] &nbsp; <strong>主题</strong>：[主题]</p>

<h3>[主题一]</h3>
<p listType="bullet" listLevel="0">[要点]</p>
<p listType="bullet" listLevel="0">[要点]</p>

<!-- 有多个发言人或子主题时，用单栏色块区分 -->
<columns>
  <column columnBackgroundColor="#EBF2FF">
    <h4>[发言人 / 子主题]</h4>
    <p listType="bullet" listLevel="0">[要点]</p>
    <blockquote>「[原话]」</blockquote>
  </column>
</columns>

<!-- 行动项直接用 todo list -->
<h3>后续行动</h3>
<p listType="todo" listLevel="0" checked="0">[行动项]</p>
```

### 会议纪要模板

```xml
<h2>会议纪要</h2>
<p><strong>主题</strong>：[主题] &nbsp; <strong>时间</strong>：[时间] &nbsp; <strong>参会</strong>：[人员]</p>

<!-- 按议题分章节 -->
<h3>[议题一]</h3>
<p listType="bullet" listLevel="0">[讨论要点]</p>

<!-- 多人发言、观点有明显差异时，用单栏色块区分 -->
<columns>
  <column columnBackgroundColor="#EBF2FF">
    <h4>[发言人]</h4>
    <p listType="bullet" listLevel="0">[观点]</p>
    <blockquote>「[原话]」</blockquote>
  </column>
</columns>

<!-- 结论内容多时用绿色块，少时直接写 -->
<columns>
  <column columnBackgroundColor="#E8FCEF">
    <h4>结论 / 决策</h4>
    <p listType="bullet" listLevel="0">[决策]</p>
  </column>
</columns>

<h3>行动项</h3>
<p listType="todo" listLevel="0" checked="0">[任务]（负责人：XX，截止：XX）</p>
```

### 课堂笔记模板

```xml
<h2>课堂笔记</h2>
<p><strong>课程</strong>：[课程名] &nbsp; <strong>时间</strong>：[时间]</p>
<h3>核心知识点</h3>
<h4>[知识点一]</h4>
<p>[核心概念与讲解]</p>
<h3>重点强调</h3>
<p listType="bullet" listLevel="0">[重点1]</p>
<h3>作业与任务</h3>
<p listType="todo" listLevel="0" checked="0">[作业]（截止：XX）</p>
```

### 知识分享模板

```xml
<h2>知识分享记录</h2>
<p><strong>主题</strong>：[主题] &nbsp; <strong>分享人</strong>：[姓名]</p>
```

### 其他专业场景

- **庭审**：按庭审程序（庭前准备→法庭调查→举证质证→法庭辩论→最后陈述）结构化整理
- **病例**：按主诉→现病史→诊断→治疗方案→医嘱结构化整理
- **新闻采访**：按 5W1H 要素 + 精彩引用 + 后续跟进整理
- **电话录音**：按通话目的→讨论内容→承诺事项→后续行动整理
- **采访记录**：按话题→核心观点→精彩引用→后续跟进整理
- **培训课程**：按课程大纲→知识点→实践练习→课后作业整理
- **口述转文章**：按引言→章节→结论结构化整理，将口语转为书面语

## Todo 提取规则（通用）

从转写中识别任务时，判断标准：

1. **明确任务**：含动作动词（完成、提交、准备、跟进）+ 时间限定或责任人
2. **承诺事项**：「我会/我们将...」「XX时候给您...」
3. **学习任务**：「需要学习/实践/应用...」

提取格式：
```
<p listType="todo" listLevel="0" checked="0">[任务描述]（负责人：XX，截止：XX）</p>
```

## 停止条件

以下情况停止循环：
- 用户明确说「停止」「结束总结」「stop」
- 连续 3 轮未检测到新的转写内容（内容无变化）
- 音频状态变为 `stopped` 且已完成最终总结

## 错误处理

| 错误 | 处理方式 |
|------|---------|
| `NO_ACTIVE_EDITOR_WINDOW` | 提示用户打开笔记，等待 30 秒后重试 |
| `WEBSOCKET_NOT_CONNECTED` | 等待 10 秒后重试获取转写 |
| `EDITOR_NOT_READY` | 等待 2 秒后重试写入 |
| `BLOCK_NOT_FOUND` | 重新 `get_note_outline` 刷新 ID 后重试 |
| `DOCUMENT_READ_ONLY` | 告知用户笔记为只读，停止写入但继续读取 |
