# 核心技术文档与笔记 — 详细流程

本文档描述在编码或代码审查过程中，如何识别核心技术、创建/编辑技术笔记并**写入 WPS 笔记**保存。与编码助手（coding-assistant）及子 skill review-notes 共用此流程。

---

## 0. 何时自动启动写入 WPS 笔记

当用户提到以下**触发用语**或**场景**时，自动启动本流程并**读取与写入 WPS 笔记、生成技术文档**：

- **编码工具 + 关键词**：用户使用 **Cursor、Codex、Claude Code、AS code** 等编码工具进行**编写、审查或优化代码**时，且用户提到「**架构**」「**设计图**」「**核心方法**」「**关键技术**」时，自动读取当前文件/工程上下文并写入 WPS 笔记，生成或更新技术文档。
- **触发用语**：审阅代码、技术文档、总结关键代码、记录关键技术点、记入笔记、生成技术文档、把这段记入笔记、记一下这个实现、整理成笔记 等。
- **场景**：使用上述编码工具编写或审查代码且涉及「核心技术」（如 Android、iOS、JNI/Rust、文档/资源/网络等）；代码或注释中出现「生成技术文档」「核心代码」「关键实现」「技术要点」。
- **优化完成后**：用户**优化代码**或**优化类**完成后，若该类或该段代码的**注释**中出现「技术文档」「总结」「笔记」等相关字样，则自动启动并写入笔记；执行时仍按先 list_notes、先查后编、标题规范、等待用户指令直接辅助等既有规则。

执行时使用 list_notes、create_note、insert_block、replace_block、insert_image、list_tags 等完成创建与保存；写完后执行**自动整理笔记格式**（见第 5 节）。**涉及的核心代码**可从注释内关键字、用户复制的代码块、剪切板中的代码块、本文件选中代码或指定函数体等方式获取并写入笔记（详见第 3 节）。

---

## 0.1 笔记标题规范

生成的笔记标题须**简洁、专业**，且与**当前类或当前目录强相关**：

- **优先**：当前类名（如 `StaticResourceInterceptor`、`OfflineNative`、`ViewModel` 等）。
- **或**：当前目录/类名（如 `interceptor/StaticResourceInterceptor`、`offlineCore/OfflineNative`）。
- **避免**：冗长描述式标题。

创建笔记时 create_note 的 title 参数即按上述规则填写。

---

## 0.2 完整笔记结构（必备二级标题）

生成的 WPS 笔记**必须完整**，且**必须包含**以下 **7 个二级标题**（顺序可微调，内容须填实）：

| 二级标题 | 内容要求 |
|----------|----------|
| **核心技术** | 用 1～2 段概括当前模块/类采用的核心技术；**须配图**：优先使用 **generate_image**（WPS 笔记）根据核心技术描述生成图片，再用 **insert_image** 插入；或使用 mermaid 图、或从官网/掘金/维基抓图后 insert_image。 |
| **核心代码** | 入口方法或关键函数完整代码块（从注释关键字、用户复制、剪切板、选中代码或指定函数获取），用 `<codeblock lang="...">` 写入。 |
| **关键技术点** |  bullet 或短段落列出关键技术点（如 JNI 桥接、异步、空安全、MIME 推断等）。 |
| **核心类和职责** | 列出涉及的核心类/模块及其职责（如 StaticResourceInterceptor、OfflineNative、ResponseUtils 及各自职责）。 |
| **调用链** | **必须**用 **mermaid** 格式（flowchart 或 sequenceDiagram）书写调用关系；**须配图**：优先使用 **generate_image** 根据调用链描述生成示意图，再用 **insert_image** 插入到该节；可同时保留 mermaid 代码块。 |
| **架构概览** | 模块在整体架构中的位置、与上下游的关系；**须配图**：优先使用 **generate_image** 根据架构描述生成架构图，再用 **insert_image** 插入；或使用 mermaid、或从官网/掘金/维基获取架构图后 insert_image。 |
| **注意事项** | 使用注意、边界情况、失败处理、最佳实践等。 |

**架构、核心技术、调用链**涉及图示时：**优先使用 WPS 笔记的 generate_image** 根据对应描述（调用链描述、架构描述、核心技术描述）生成图片，再用 **insert_image** 插入笔记；若 generate_image 不可用，则用 mermaid 或从官网/掘金/维基 insert_image。上述三处至少一处须含具体图示，不可仅文字描述。

---

## 0.3 用户新增标题时的处理

当用户在已生成的笔记里**新增二级标题或其他小标题**（如「扩展点」「依赖说明」「测试建议」等）时：

- **根据用户诉求补充内容**：通过用户当前输入、上下文或自然语言意图，推断该新标题下应写什么，并**在该标题下插入相应段落或列表**（用 insert_block 在对应标题后插入内容）。
- 若用户未明确说明内容，可根据当前文件/工程、笔记已有内容合理推断并补写 1～2 段或若干 bullet。

---

## 0.4 持续上下文与主动更新

- **上下文**：以用户**当前打开的文件或当前工程**为上下文；该上下文在用户**关闭本文件或关闭本工程**前持续有效。
- **主动更新**：在用户**未主动关闭当前笔记**期间，agent 应**主动、周期性地更新**该笔记：
  - **约 1 分钟后**或适当时机（如当前打开文件/工程有变化、或会话中有新讨论），**主动**刷新并更新该笔记（如补充新小节、更新调用链、同步核心代码、补全架构概览或核心技术图等）。
  - 持续此行为**直到用户主动关闭当前笔记**为止。用户关闭该笔记后，该「当前笔记」上下文结束，不再对该笔记做主动更新。
- **标签**：笔记的标签宜与当前类或当前目录相关，并随主动更新与当前文件/目录保持对齐。

## 1. 识别「核心技术」

- **方式一（用户标注）**：类或函数注释中出现「**生成技术文档**」或同义表述时，将该处涉及的技术点纳入梳理并生成/更新笔记。
- **方式二（复杂场景）**：代码涉及多线程/协程、JNI/NDK、复杂状态与导航、性能优化、安全与加密等时，视情况纳入并生成技术文档。

---

## 2. 先查后编（不重复新建）

1. **先 list_notes** 查看已有笔记，根据**当前类名或当前目录/类名**（见 0.1）在结果中匹配是否已存在对应笔记（标题包含类名或目录/类名子串即视为匹配）。
2. **若已存在匹配笔记**：调用 **read_note**（及 **get_note_outline**）获取该笔记的 note_id 与结构；**不新建**，后续所有写入与编辑**均在该笔记上**进行（insert_block、replace_block、insert_image 等）。
3. **若不存在匹配笔记**：再调用 **create_note** 创建新笔记；**标题**按第 0.1 节（简洁、专业、与当前类或当前目录强相关）。创建后用 **insert_block** / **replace_block** 写入内容。
4. 内容**必须**按第 0.2 节「完整笔记结构」包含 7 个二级标题：**核心技术**（配图）、**核心代码**、**关键技术点**、**核心类和职责**、**调用链**（mermaid）、**架构概览**（配图）、**注意事项**。**最终成果须保存在 WPS 笔记中**。

## 2.1 架构 / 核心技术 / 调用链的配图：generate_image + insert_image

笔记中涉及**架构**、**核心技术**、**调用链**的图示，按以下流程优先生成并插入：

1. **撰写文字描述**：根据当前模块的调用链、架构层次、核心技术点，写一段简明的图示描述（例如：「流程图：WebView → RequestInterceptor → StaticResourceInterceptor.getStaticResource → OfflineNative.getStaticResourceExt → JNI → Rust → byte[] → ResponseUtils → WebResourceResponse」或「架构图：Android 层、JNI 层、Rust 离线层三层结构」）。
2. **调用 generate_image**：使用 **WPS 笔记的 generate_image**（若当前环境提供），将上述描述作为 prompt 传入，生成对应示意图（调用链图、架构图、技术原理图等）。
3. **调用 insert_image**：将 generate_image 返回的图片（或可访问的 URL/本地路径）通过 **insert_image** 插入到笔记中对应二级标题下（如「调用链」「架构概览」「核心技术」）。

若 **generate_image** 不可用，则采用 mermaid 代码块或从官网/掘金/维基百科抓取关键图后 insert_image 插入。

## 2.2 调用链用 mermaid 格式

笔记中的「调用链」或「调用关系」须用 **mermaid** 格式书写，便于在支持 mermaid 的编辑器中渲染；并可同时用 2.1 的 generate_image 根据该调用链生成示意图后 insert_image 插入：

- **flowchart**：`flowchart LR` 或 `flowchart TD`，节点为类/方法名，箭头表示调用。示例：`A[WebView] --> B[RequestInterceptor] --> C[getStaticResource] --> D[JNI]`。
- **sequenceDiagram**：`sequenceDiagram` 中 `participant` 声明参与方，`->>` 表示调用，可标注方法名。

## 2.3 等待用户指令并直接辅助

确定**当前笔记**（已有或新建）后，**等待用户后续意图**，并**直接在该笔记上完成**，无需用户再单独发「对某笔记做某操作」的指令：

- **插入图片**：用户提供图片或说「加图」「插张图」等 → 直接对该笔记调用 **insert_image**。
- **插入核心代码**：用户选中代码、说「助手核心代码」或告知已复制到剪贴板 → 直接在该笔记核心代码区 **insert_block** / **replace_block** 写入代码块。
- **修改标题**：用户说「改标题」「改下标题」或给出新标题 → 直接 **replace_block** 或 **update_block_attrs** 修改该笔记标题 block。
- **其他操作**：全局替换、整理格式、加标签等 → 均直接针对该笔记执行，无需用户指明「对哪篇笔记」。

---

## 3. 核心代码：来源与写入

**核心代码**可从以下任一种方式获取（满足其一即可）：

- **注释内关键字**：代码注释中出现「**核心代码**」「**关键实现**」「**技术要点**」「生成技术文档」等关键字时，读取关键字所在行、所在代码块或对应函数体作为核心代码。
- **用户复制的代码块**：用户在本文件中选中并复制一段代码后告知（如「已复制」「这段是核心代码」），将该段作为核心代码写入笔记。
- **剪切板中的代码块**：用户将代码复制到剪切板后告知（如「剪切板里有」「粘贴了核心代码」），以剪切板内容作为核心代码；若无法直接读取剪切板，则从当前打开文件中提取最相关的函数/代码块作为备选。
- **本文件选中代码**：用户在本文件中选中的代码。
- **指定函数**：用户指定「某个函数」或通过注释/上下文可识别的单个函数时，取该**函数的完整函数体**作为核心代码。

写入方式：使用 **insert_block** 或 **replace_block** 在笔记的「核心代码」对应 block 处插入 `<codeblock lang="...">`（按实际语言填 kotlin、java、swift、rust 等）包裹的代码。若该区域尚为空，先插入占位段落再替换为代码块，或直接插入代码块。

**配图**：对上述核心代码段可调用 **insert_image** 插入配图。技术原理图可优先用 **generate_image** 根据描述生成后 insert_image；或从官网、掘金高赞、维基百科等获取关键图后 insert_image 插入笔记。

---

## 4. 笔记内全局替换（标题、人名、关键技术术语）

当用户**手动修改**了笔记中某一处的**标题、人名或关键技术术语**时，应在**同一篇笔记内**做**全局查找并替换**，使全文表述一致。步骤：确认新表述 → **search_note_content**（或 **read_note**）查找旧表述所在 block → 对每个 block 使用 **replace_block** 或 **batch_edit** 替换。

---

## 5. 自动整理 WPS 笔记格式

在**创建或编辑**笔记后，对当前笔记执行一次格式整理：统一标题层级（h1→h2→h3）、列表与段落格式，避免孤段与多余空行；核心代码区使用 `<codeblock lang="...">` 并设正确 lang。通过 **get_note_outline** 查看结构，用 **replace_block** 或 **update_block_attrs** / **batch_edit** 修正。

---

## 6. 笔记完成后的标签

调用 **list_tags** 查看用户已有标签；根据笔记主题与内容，参考用户已有标签的命名风格生成核心标签；若平台支持则为笔记添加标签，否则输出「建议标签」列表供用户手动添加。

---

## 7. 与 WPS 笔记协作的智能与便捷规则

**当前笔记的智能解析**（用户说「加图」「改标题」「插入核心代码」等未指明哪篇笔记时）：

1. 优先调用 **get_current_note**：若用户正在 WPS 中打开并编辑某篇笔记，则取该 note_id 作为「当前笔记」。
2. 若无当前笔记或调用失败：沿用本会话中已确定的「当前笔记」（即本次 list_notes/read_note 或 create_note 得到的那篇）。
3. 若会话中尚未确定：再按 **list_notes** + 当前打开的文件/类名或目录匹配（标题包含类名或目录子串即视为匹配），取匹配到的第一篇；无匹配再 create_note 新建。
4. 用户说「这篇」「当前笔记」「就这个」时，直接沿用已确定的 note_id，无需再次 list_notes。

**编辑前刷新 block_id**：对某笔记进行 **replace_block** / **insert_block** / **batch_edit** 前，若尚未在本轮获取过该笔记的 outline，或距上次 **get_note_outline** 已较久，则先调用 **get_note_outline(note_id)** 获取最新 block 列表与 id，再执行编辑，避免 block_id 过期导致 BLOCK_NOT_FOUND。

**编辑后同步**：每次对该笔记执行 insert_block、replace_block 或 batch_edit 后，调用 **sync_note(note_id)**，使修改在 WPS 客户端尽快可见并持久化。

**错误处理**：若出现 BLOCK_NOT_FOUND 或类似「block 不存在」：先 **get_note_outline** 刷新该笔记的 block 列表，用新的 block_id 重试操作；若仍失败，简要告知用户并建议在 WPS 中手动检查该笔记。

**合并操作减少往返**：同一笔记内有多处插入或替换时，优先使用 **batch_edit** 在一次调用中完成 delete/replace/insert，减少多次单独调用。

**主动提示**：在完成一次笔记创建或更新后，可简短提示用户下一步可选操作，例如：「已保存。可以说「加图」「助手核心代码」或「改标题」继续编辑当前笔记。」

**用户新增标题**：若用户在笔记中新增了二级标题或其他小标题，根据用户诉求或上下文**在该标题下补充相应内容**（见第 0.3 节）。**主动更新**：在用户未关闭当前笔记期间，约 1 分钟后或适当时机主动刷新并更新该笔记，直至用户关闭当前笔记（见第 0.4 节）。
